Wiring & PID

How the motor controller is built, and how the control law works.

The hardware

Arduino UNO + L298N + gearmotor with encoder

A single-direction velocity controller: the Arduino reads motor speed from a quadrature encoder, runs the PID at 40 Hz, and drives the motor through an L298N H-bridge with a PWM signal. Schematic derived from the firmware firmware/v1.ino.

Signal flow: you set a target RPM → the PID (in the UNO) sets a PWM duty on ENA → the L298N drives the motor → the encoder measures real speed → that feeds back to D2/D3, closing the loop.

Power the motor from the L298N's own 12 V supply, not the Arduino's 5 V. Always tie all grounds together. Adjust COUNTS_PER_OUTPUT_REV in the firmware to your encoder + gearing so reported RPM is correct.

Before you power up

Build & safety checklist

  1. Remove the L298N ENA jumper so D5's PWM actually controls the speed.
  2. Keep the 5V-EN jumper installed and leave the L298N +5V screw terminal empty — the Arduino runs from USB.
  3. Tie every ground together (Arduino, L298N, and the 12 V supply). This is the single most important connection — without a shared ground nothing works reliably.
  4. Encoder A → D2, B → D3. Those are the UNO's only external-interrupt pins; the encoder ISR needs them.
  5. Bench-test with the shaft free and NO load. The L298N handles ~2 A; a stalled motor pulls ~8.5 A and will destroy the driver. The 3 A fuse is your backstop — this v1 firmware has no stall cutoff.
  6. Serial Monitor / Serial Plotter at 115200 baud, "Newline". Type an RPM (e.g. 300) to run; s to stop; p/i/d <value> to tune live — or connect straight from the Playground's Hardware mode.
The control law

Feed-forward + PID, once every 25 ms

Each tick, the Arduino reads how many encoder counts arrived since last time, converts that to RPM, and computes a PWM output. Here's the whole loop — the simulator runs this exactly:

// dt = 0.025 s (40 Hz)
rpmRaw     = |counts| / 70 / dt * 60          // encoder → RPM
rpm        = S*rpm + (1-S)*rpmRaw             // EMA smoothing, S = 0.6
error      = target - rpm
derivative = -(rpm - rpmPrev) / dt            // on the MEASUREMENT
feedForward = target>0 ? (offset + slope*target) : 0

output = FF + Kp*error + integral + Kd*derivative
if (output > 0 && output < 255) integral += Ki*error*dt   // anti-windup
integral = clamp(integral, -255, 255)
output   = FF + Kp*error + integral + Kd*derivative        // recompute
output   = clamp(output, 0, 255)
if (target <= 0) { output = 0; integral = 0 }              // stop resets I

Feed-forward does most of the work

FF = offset + slope·target pushes out the PWM the motor needs before any error is measured. Match it to the motor and feed-forward alone hits the target.

Integral erases the leftover

Whatever FF and P miss, the I term accumulates until steady-state error is zero. Under load it climbs to supply the extra push.

Anti-windup freezes I at the rails

Integration only happens while 0 < output < 255. When the output saturates, the integral stops growing, so it can't wind up and overshoot on recovery.

Derivative on the measurement

Using −(Δspeed)/dt instead of the error means a step change in target causes no derivative "kick" — D only reacts to real speed changes, damping them.

Smoothing tames the encoder

At 40 Hz each count is ≈34 RPM, so the raw reading is a staircase. The EMA weight S calms it — at the cost of a little lag.

Why 25 ms?

40 Hz is fast enough to catch the motor's ~0.12 s response, slow enough that each tick gathers plenty of encoder counts for a clean speed estimate.

Talking to it

Serial protocol · 115200 baud

Newline-terminated. Telemetry streams ~40×/sec; commands are single lines.

SP:<target>,RPM:<speed>,OUT:<pwm>
// with 'r' debug on: SP:400,RPM:398,OUT:110,dCounts:12,A:1,B:0
CommandEffect
<number>set target RPM (0–1000)
sstop the motor (target 0, clears the integral)
p <v>set Kp
i <v>set Ki
d <v>set Kd
rtoggle raw encoder debug fields
The stock firmware has no command for feed-forward or smoothing — those are compiled-in constants, which is why the app disables those sliders in Hardware mode. Adding o/f commands is on the roadmap.
